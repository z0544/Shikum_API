# תוכנית שיפור UI/UX — מערכת שיקום (Client)

**גרסה:** 1.0 · **תאריך:** 2026-07-17 · **מחבר:** Claude (Product Spec Writer)
**בסיס:** ביקורת קוד מלאה של `client/src` (כל 18 הקבצים) + בדיקה חיה בדפדפן: דסקטופ, מצב כהה ומובייל (375px).

---

## 1. תקציר מנהלים

המערכת פונקציונלית ועשירה ביכולות (חיפוש מדויק/AI, עמוד וריאנט, ניהול, תיעוד API, צ'אטבוט), ויש בה יסודות טובים: מצב כהה מלא, קישורים עמוקים (deep-links), טוקני עיצוב, ומיקרו-אינטראקציות בצ'אט. אבל שלוש בעיות-על פוגעות בחוויה בצורה ניכרת:

1. **רספונסיביות שבורה** — מתחת ל‑900px ה‑header מתפרק (כפתורי ניווט צפים על התוכן), ובמובייל יש גלישה אופקית של העמוד כולו. media query אחת בלבד בכל הפרויקט.
2. **נגישות (A11y) חסרה באופן שיטתי** — אף דיאלוג לא לוכד פוקוס ולא נסגר ב‑Escape; שורות תוצאה וספקים הן `div`/`tr` לחיצים בלבד (אין גישה ממקלדת); ה‑Toast — ערוץ השגיאות הראשי — לא מוכרז לקורא מסך.
3. **חוסר עקביות עיצובי** — טוקני ריווח וטיפוגרפיה מוגדרים אך כמעט לא בשימוש; צבעי hex קשיחים בלי וריאנט למצב כהה (בלוקי cURL בלתי קריאים במצב כהה); אימוג'י לצד מערכת אייקונים; דיאלוגים של הדפדפן (`confirm`/`prompt`) בתוך מערכת מעוצבת.

התוכנית בנויה בשישה שלבים: תיקוני-בזק (P0) → תשתית רספונסיבית → נגישות WCAG → קונסולידציית Design System → זרימות ומצב (state) → ליטוש צ'אט וניהול. כל ממצא מגובה בהפניית `קובץ:שורה`.

## 2. מתודולוגיה

- **ביקורת קוד סטטית** — כל קבצי `client/src` כולל `theme.css`, `AppContext`, וכל הרכיבים.
- **בדיקה חיה** — `http://localhost:3000` (build פרודקשן): חיפוש "642" (54 מק"טים), הרחבת קבוצות, בחירת וריאנט, עמוד וריאנט ייעודי, צ'אט (שאילתה + תשובת AI), מסך ניהול (שער טוקן), מסך API; בשלושה מצבים: רוחב בינוני (~790px), מצב כהה, מובייל (375px).
- **מסגרת הערכה** — עשרת ההיוריסטיקות של נילסן, WCAG 2.2 AA, ודפוסי master-detail מקובלים לכלי קטלוג פנים-ארגוניים.

## 4. עקרונות עיצוב מנחים

1. **כלי עבודה, לא אתר** — המשתמשים חוזרים אליו יומיומית; מהירות-למשימה גוברת על אפקטים. כל אינטראקציה מרכזית (חיפוש → בחירה → פרטים → ספק) חייבת לעבוד במקלדת בלבד.
2. **המסמך הוא ה‑URL** — כל מצב (שאילתה, מצב חיפוש, וריאנט נבחר) ניתן לשחזור מהכתובת; מעבר מסכים לעולם לא מוחק עבודה.
3. **פידבק לכל פעולה** — אין פעולה שמסתיימת בשקט: טעינה, הצלחה, כשל וריק מקבלים ייצוג נראה ומוכרז.
4. **טוקנים כחוק** — צבע, ריווח וטיפוגרפיה רק דרך משתני CSS סמנטיים שיש להם ערך בשני המצבים (בהיר/כהה).
5. **RTL כאזרח ראשון** — רק מאפיינים לוגיים (`inset-inline`, `margin-inline`); איי LTR מפורשים לקוד ומזהים.
6. **מובייל אמיתי** — טכנאי שטח/עובד מול מטופל עם טלפון: ניווט מקופל, master-detail כ‑bottom-sheet, מטרות מגע ≥44px.

### שלב 1 — תשתית רספונסיבית (השבוע הראשון)

1. **בניית Header מחדש (P0‑1, P0‑2)**
   - הסרת הגובה הקבוע; grid: לוגו | ניווט | פעולות.
   - ברוחב < 900px: תפריט המבורגר (Drawer מימין, RTL) או שורת טאבים שנייה מתחת ללוגו — מומלץ טאבים נגללים אופקית (פשוט יותר, בלי מודאל נוסף).
   - איתור וחיסול מקור הגלישה האופקית במובייל (min-width קשיח / תוכן שלא נשבר) — `overflow-x` ברמת containers, `min-width: 0` בעמודות grid.
2. **סולם breakpoints מוגדר**: 600 / 900 / 1200, כמשתני עזר מתועדים ב‑theme.css.
3. **דפוס Master-Detail אמיתי (P0‑4)**
   - דסקטופ: פאנל פרטים sticky (`position: sticky; top`) בעמודה השנייה, כך שהוא נראה תמיד בזמן גלילת תוצאות.
   - מובייל/עמודה בודדת: בחירת וריאנט פותחת **bottom-sheet** (או ניווט לעמוד הוריאנט הייעודי הקיים — לנצל את `#/variant/<id>` שכבר בנוי) במקום לרנדר פאנל סמוי בתחתית.
4. **כותרות קבוצה ברוחב מלא (P2‑10)** — `width: 100%` ל‑group head, יישור אחיד של הצ'יפים.

### שלב 2 — נגישות WCAG 2.2 AA (שבוע)

1. **רכיב `<Dialog>` אחוד (P0‑5)** — מבוסס `<dialog>` נטיבי או focus-trap ידני: `aria-modal`, פוקוס ראשוני, לכידת Tab, Escape, החזרת פוקוס לפותח. הסבת ארבעת המודאלים + פאנל הצ'אט אליו.
2. **סמנטיקה לחיצה (P0‑6)** — שורות וריאנט הופכות ל‑`<button>` ברוחב מלא (או `role="option"` בתוך `listbox`); שורות ספק מקבלות כפתור הרחבה אמיתי בתא הראשון עם `aria-expanded`; dropzone הופך ל‑`<button>` עם key handlers.
3. **autocomplete מקלדתי (P1‑9)** — דפוס `combobox` ARIA מלא: חצים, Enter, Escape, `aria-activedescendant`; איחוד לשני המימושים לרכיב אחד.
4. **פירורי לחם אמיתיים (P1‑6)** — `<nav aria-label="breadcrumbs">` עם קישורי hash אמיתיים, או הסרתם לגמרי (עדיף — יש כבר טאבים; פירורי לחם כפולים לניווט שטוח הם רעש).
5. **ביקורת ניגודיות** — בדיקת כל הצ'יפים/badges בשני המצבים (כלי אוטומטי + ידני), במיוחד הענבר בהיר על לבן.
6. **skip-link + `lang`/`dir` בקומפוננטות LTR** (קוד, מזהים).

### שלב 3 — קונסולידציית Design System (שבוע)

1. **אכיפת טוקנים (P2‑1)** — מעבר יזום על theme.css וה‑inline styles: כל px ריווח → `--space-*`, כל font-size → `--fs-*`; מחיקת הטוקנים אם הוחלט שלא בשימוש (לא להשאיר מערכת כפולה).
2. **טוקנים סמנטיים דו-מצביים** — `--success-bg/ink`, `--warn-bg/ink`, `--danger-bg/ink`, `--code-bg/ink` עם ערכים לבהיר ולכהה; כל hex קשיח מוחלף.
3. **רכיבי בסיס משותפים** — Button (וריאנטים: primary/ghost/danger), Chip, Card, EmptyState, ErrorState (עם retry), Skeleton, Dialog, ToastStack (תור של עד 3, `aria-live`), ConfirmDialog מעוצב שמחליף את `confirm()`/`prompt()` (P2‑2, P2‑3 — ומוסיף אישור למחיקת מילה נרדפת).
4. **ניקוי אימוג'י (P2‑5)** — 👋/📎/📄/◈ מוחלפים באייקונים מ‑`icons.tsx`; ◈ הופך ללוגו SVG.

### שלב 4 — זרימות ומצב (שבוע)

1. **שימור state בין מסכים (P1‑1)** — העלאת מצב החיפוש (query, mode, groups, expanded, selected, scroll) ל‑AppContext או `keepAlive` render (הסתרה ב‑CSS במקום unmount). קריטריון קבלה: חיפוש → כניסה לוריאנט → חזרה = אותן תוצאות ואותה גלילה.
2. **URL כמצב מלא (P2‑9)** — `#/search/<q>?mode=ai&match=…&field=…`; חיפוש מוזרק מכבד את המצב; שיתוף כתובת משחזר בדיוק.
3. **שקיפות ה‑fallback ל‑AI (P1‑4)** — כשהחיפוש נופל ל‑AI: הטוגל עובר ויזואלית ל"חיפוש חכם" עם banner "לא נמצאו תוצאות מדויקות — הצגת תוצאות חכמות · [חזרה למדויק]". ה‑state היחיד הוא `ranMode`.
4. **ErrorState בפאנל הפרטים (P1‑3)** — "לא ניתן לטעון את הוריאנט · נסה שוב" במקום `null`; אותו דפוס ל‑refresh בניהול (P1‑10) עם banner שגיאה.
5. **עימוד תוצאות (P1‑8)** — "מציג 54 מתוך N" + כפתור "טען עוד" (או וירטואליזציה אם N גדול); כפתור "חזרה למעלה" צף בגלילה ארוכה.
6. **חיווי חיתוך בטבלאות preview** — "מציג 100 מתוך N שורות" (P1‑10).

### שלב 5 — ליטוש צ'אט וניהול (3–4 ימים)

1. **התמדת שיחה** — שמירת `msgs` ב‑sessionStorage; שחזור אחרי רענון; "שיחה חדשה" מנקה במפורש.

- **ניהול** — הוסרו הספירות הקשיחות המיושנות מהטאבים. (`AdminView.tsx`)

**נדחה משלב 0 (דורש אישור בעל הנתונים):** המרת ערכי sentinel (999/99999 → "ללא הגבלה") ועיצוב מטבע (₪) ל-`amount` — נבדק בקוד השרת ואין שם התייחסות ל-999/99999 כ"ללא הגבלה", ומשמעות `amount` תלויה בשדה `amountTypeRaw` (₪ מול אחוז). לכן המרה אוטומטית עלולה להציג נתון שגוי. יש לאשר את הסמנטיקה מול בעל הנתונים לפני היישום.
